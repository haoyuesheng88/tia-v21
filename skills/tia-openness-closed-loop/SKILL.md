# TIA Openness Closed Loop

Use this skill for `TIA Portal V21` work that involves:

- Attaching to a live TIA process with `Openness`
- Injecting or updating `SCL` in `S7-1200` CPUs
- Injecting or updating `SCL` in `S7-1200G2` CPUs such as `CPU 1212FC`
- Building a PLC/HMI closed loop with absolute addresses
- Configuring `KTP1200 Basic PN` classic HMI and `Unified Comfort` together
- Working with `TP1200 Comfort` classic HMI and `VBScript`
- Importing `WinCC Unified V21` global JavaScript modules into a live project
- Building example process logic such as a baghouse pulse dust collector
- Repairing broken HMI tags caused by symbolic DB tags, bad connections, or classic/Unified model differences

## Quick Rules

- Prefer `Openness` first. Use UI automation only when the object model does not expose the needed action.
- For mixed PLC/HMI loops, prefer absolute addresses over optimized DB symbolic tags:
  - `%M10.0` start pulse
  - `%M10.1` stop pulse
  - `%M10.2` run latch
  - `%M10.3` lamp feedback
  - `%MW20` set value
  - `%MW22` actual value
- Keep the process loop in PLC logic, not in HMI scripts.
- For `WinCC Unified`, use `JavaScript`.
- For `WinCC Unified`, prefer importing script modules through `HmiScriptModuleComposition.Import(...)` instead of typing large scripts through the UI.
- For `TP/Comfort` classic panels, prefer `VBScript` plus PLC-owned process logic.
- For `KTP1200 Basic PN`, do not assume Unified behavior or script model.
- When the user asks for an industry example, prefer implementing it as an isolated `FB + instance DB + OB call` so the existing program is disturbed as little as possible.

## Verified Closed-Loop Pattern

Use an `SCL` block or equivalent logic like this:

```scl
IF %M10.0 THEN
    %M10.2 := TRUE;
    %M10.0 := FALSE;
END_IF;

IF %M10.1 THEN
    %M10.2 := FALSE;
    %M10.1 := FALSE;
END_IF;

%Q0.0 := %M10.2;
%M10.3 := %Q0.0;
%MW22 := %MW20;
```

This gives:

- HMI input -> `%M10.0/%M10.1/%MW20`
- PLC processing -> `%M10.2/%Q0.0/%MW22`
- HMI feedback -> `%M10.3/%MW22`

## Workflow

1. Attach to the live TIA process and confirm the project path, target CPU, and HMI names.
2. Inspect the real device inventory before editing. Do not trust spoken model names when the project says otherwise.
3. Inject or update PLC `SCL` first.
4. Rebuild PLC tag tables to match the absolute address scheme.
5. Configure `Unified` HMI tags by `Openness` if possible.
6. If the request is for a process example, create a self-contained block pair such as `FB_*` + `DB_*`, then connect it from `Main`.
7. For `Unified` script work, export one real module first or use a known-good import package format, then import the module into `HMI_RT_2`.
8. For classic `KTP1200 Basic PN` or `TP1200 Comfort`, confirm whether the connection exists in UI even if `Openness` reports `0` connections.
9. If the classic HMI object model blocks creation, use the UI only to establish or confirm the connection, then continue with export/import driven edits where possible.
10. Compile and re-check each layer:
   - PLC compile
   - Unified tags/connection readback
   - Unified script module readback
   - Classic HMI tag table export or compile messages

## Classic vs Unified

### Unified

- Connection model is explicit and works well with `Openness`.
- Tags can be created with absolute access.
- Global script modules can be imported by directory using `HmiScriptModuleComposition.Import(...)`.
- Exported module structure is a pair of files:
  - `ModuleName.hmi.yml`
  - `ModuleName.hmi.js`
- The `.hmi.yml` file maps the module name to the `.hmi.js` file.
- The `.hmi.js` file should preserve the autogenerated header comment.
- Use `JavaScript` for runtime logic and UI behavior.

### Classic KTP1200 Basic PN

- The connection may be visible in the UI but not enumerate through `HmiTarget.Connections`.
- Classic tag export/import uses XML content even when the file extension is `.xlsx`.
- When debugging classic tags, export the tag table and inspect the XML payload.
- If symbolic DB tags are unstable or invalid, switch the design to absolute `M/MW` addresses.

### TP1200 Comfort / Classic Comfort Panels

- `VBScriptFolder` exists under the classic HMI software object.
- `VBScriptComposition` supports `Import(FileInfo, ImportOptions)` and `Find(name)`.
- Prefer PLC-driven sequencing and use `VBScript` for operator actions, tag writes, and UI feedback.

## Known Pitfalls

- `DB_HMI.Start_PB` style symbolic references can break HMI binding when optimization or symbolic resolution is inconsistent.
- A classic HMI connection can appear valid in the editor while `Openness` still returns `COUNT|0`.
- UI-created classic HMI rows may not persist all fields until explicitly saved.
- Mixed Chinese/English connection names are common. Preserve the exact connection name from export, for example `HMI_连接_1`.
- Do not assume exported classic HMI files are real Excel workbooks just because the extension is `.xlsx`.
- `Unified` script import is format-sensitive. Start from a real exported module when unsure.
- Importing a `Unified` script module does not automatically bind screen events or tasks.
- Exporting a block from a `1212FC` project may fail noisily through `Openness`; avoid depending on block export when direct source injection is enough.

## Validation Checklist

- PLC compile succeeds
- `%Q0.0` follows `%M10.2`
- `%M10.3` mirrors `%Q0.0`
- `%MW22` mirrors `%MW20`
- Unified HMI has one valid PLC connection and six working tags
- Unified HMI contains the expected global script module, for example `ClosedLoopModule`
- Exporting the imported Unified module reproduces the expected functions
- Classic HMI variable table references the correct connection and no stale DB symbols remain
- If a `CPU 1212FC` process example was requested, confirm the example blocks exist and the expected I/Q/MW addresses are present
